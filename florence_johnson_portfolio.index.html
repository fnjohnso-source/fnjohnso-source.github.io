DentalOne Partners had grown rapidly through acquisition, absorbing dozens of independent dental practices that each ran billing their own way — different systems, different workflows, different clearinghouse setups, different denial patterns. The org needed a single person who could bridge the clinical, billing, and technical worlds — and own the program end to end.

AI-Assisted Documentation Workflow
Solventum (3M Healthcare)  ·  CHP Data Integration & Reporting  ·  Jira · Microsoft Copilot  ·  2024
The Situation — Why This Was Needed

During the CHP (Child Health Plus) eligibility file onboarding for NYS Medicaid managed care, the team was managing 9 health plans simultaneously with 5 active integration workstreams per payer. Documentation — requirement summaries, test cases, and sprint release notes — was being written from scratch each sprint, consuming 4–5 analyst hours per cycle. I designed a Copilot-assisted prompt workflow that generated structured first drafts directly from Jira epics and acceptance criteria, cutting draft time to under 60 seconds while keeping analysts firmly in the review and ownership seat.

How the Workflow Operated
📥
Step 1 — Input
Jira epic, user stories, and acceptance criteria pasted into Copilot with a structured prompt template (one per doc type)
🤖
Step 2 — Copilot Draft
Structured first draft generated in under 60 seconds — correct sections, business context pulled from Jira, gaps flagged as [ANALYST REVIEW NEEDED]
✅
Step 3 — Analyst Review
Analyst corrects terminology, adds edge cases, validates scope, and signs off. Analyst owns the final output — Copilot produced the scaffold.
Three Documentation Templates Built & Deployed
Template 1 of 3 Requirement Summary PMED-1142 · CHP-004
Copilot Prompt Approach

Fed Jira epic CHP-004 (Child Health Plus Eligibility File Onboarding) + acceptance criteria. Copilot generated a six-section structured document: Business Context, Scope, Out of Scope, Key Assumptions, Business Rules, Open Questions — with gaps pre-flagged.

Analyst Edits That Mattered
  • Rewrote Business Rule BR-3 (coverage date overlap logic) — Copilot's version didn't account for same-day MMC/CHP overlap edge case
  • Added BR-5 (audit log requirement) entirely — not present in Jira acceptance criteria; analyst judgment call
  • Added Open Question #1: whether DOH requires separate or combined CHP/MMC submission files
  • Confirmed age-eligibility ceiling (under 19) needed a hard validation reject, not a warning flag — escalated to DOH for confirmation
Template 2 of 3 Test Case Description PMED-1156 · CHP-004-S2
Copilot Prompt Approach

Fed Jira story PMED-1156 (CHP Age Validation, Business Rule 2). Copilot produced a fully executable test case with Test Objective, Pre-conditions, Test Data table (5 member records with DOB/eligibility date/expected outcomes), numbered Test Steps with expected results, Pass Criteria, and Edge Cases — written for offshore QA execution without US-side support.

Analyst Edits That Mattered
  • Added Test Data row CHP-TEST-003 (boundary: exactly 18 years) — Copilot missed the boundary condition entirely
  • Added audit log field requirements to Test Steps 4–5 (file name, receipt timestamp, record count, pass/reject breakdown)
  • Confirmed steps were executable by offshore QA team — validated through a dry-run with the QA lead
Template 3 of 3 Sprint Release Notes Sprint 22 · Build 3.14.0
Copilot Prompt Approach

Fed the list of closed Jira tickets from Sprint 22 (PMED-1142, 1143, 1149, 1153, 1155, 1156, 1158, 1161, 1162). Copilot generated client-facing release notes grouped by feature area — New Features, Enhancements, Bug Fixes, Known Limitations — written for MCO client contacts and DOH liaisons, not engineers.

Analyst Edits That Mattered
  • Rewrote Bug Fix row 3 (PMED-1155, mid-month plan transfer logic) — Copilot's wording was technically accurate but confusing to MCO client contacts
  • Added Known Limitation #3 (exception queue combined CHP/MMC view, filter planned Sprint 23) — surfaced from CHP pilot client feedback, not in Jira
Outcome & How to Frame This
What Copilot Did
  • Produced the correct structure and section headers instantly
  • Pulled business context from Jira epics accurately
  • Pre-flagged documentation gaps as [ANALYST REVIEW NEEDED]
  • Formatted test data tables and numbered steps correctly
What the Analyst Did
  • Added edge cases and boundary conditions Copilot missed
  • Identified requirements not captured in Jira (BR-5, Open Question #1)
  • Rewrote client-facing language for MCO audience clarity
  • Owned all final accuracy, approval, and distribution
The Right Way to Frame This

"I built prompt templates tailored to each doc type — requirement summaries, test cases, release notes — and used Copilot to generate structured first drafts from Jira artifacts. Analysts reviewed and owned the output. We recaptured 4–5 hours per sprint." This is a human-in-the-loop, responsible AI adoption story — not automation for its own sake, but precision tooling that made analysts faster without removing their judgment from the process.
